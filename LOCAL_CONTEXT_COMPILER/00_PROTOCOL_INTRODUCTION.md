## WHAT THIS IS // 01

This corpus specifies a Local Context Compiler: a governed pipeline that ingests high-entropy operator material (notes, transcripts, documents, repositories, calendars) and emits bounded, inspectable context packs for language-model runtimes.

The compiler is treated as a specification problem in the OpenIP sense: stable text-defined procedures, explicit artifact types, declared coordinates, and conformance claims that can be checked. The runtime remains replaceable.

This corpus is a **corpus-level projection** of the OpenIP kernel in `the-corpus/`: it adapts the kernel’s structural obligations to the specific problem of context compilation, and therefore inherits ShareAlike licensing. The derivative relationship is intentional and explicit.

## THE PROBLEM // 02

Assistants at scale face an incentive gradient toward retention and monetization. The lever is continuity; the raw material for continuity is personal context. Vendor-owned memory pipelines accumulate context silently, derive latent profiles, and convert dependence into default capture.

The technical pipeline (logging → inference → embeddings → retrieval → compaction) is not the issue; custody is. When the pipeline is opaque or vendor-locked, personalization becomes extraction by default.

## THE SOLUTION // 03

The alternative is an operator-owned, inspectable compiler that externalizes continuity into user-controlled artifacts and procedures.

The core deliverable is a system that produces three explicit outputs:

| Output | Purpose |
|--------|---------|
| **Compiled context** | What a runtime sees; bounded, inspectable, editable |
| **Explanation trace** | Why each item was selected; evidence-linked selection, not “mind-reading” |
| **Candidate memory updates** | Proposed long-term state changes; deliberate, never silent |

Continuity survives runtime swaps because it is stored as artifacts, not as a vendor profile. Derived structure (summaries, embeddings, indices) is treated as rebuildable cache, not as truth.

## THE COMPILER STANCE // 04

Compilation, not assistance, is the framing. A context pack is a compiled artifact: it declares what category it is, what inputs were permitted, what boundaries were enforced, and what acceptance criteria define “done.”

The compiler’s job is not to become intimate; it is to make selection inspectable and termination explicit. The operator retains custody: promotion into long-term state requires explicit ratification.

## SCOPE AND BOUNDARIES // 05

This corpus governs:

- artifact types and their coordinates (what kind of thing each file is);
- compaction and packing procedures (named transforms with acceptance criteria);
- epistemic status and promotion rules (settled vs provisional);
- boundary enforcement (explicit separation between categories of context);
- conformance claims (what can be tested, and how).

This corpus does not:

- prescribe domain content or recommendations;
- define a hosted service;
- grant a vendor access to unbounded state;
- equate “memory” with silent accumulation.

## HOW TO READ THIS CORPUS // 06

Start with the Manifest for scope and license. Then proceed through Structural Obligations, Coordinate System, Compiler Model, Artifact Types, Procedures, and Conformance.

This corpus is intended to become executable in the OpenIP sense: procedures defined in text first, implemented in code later, and verified by fixtures and regression checks.
